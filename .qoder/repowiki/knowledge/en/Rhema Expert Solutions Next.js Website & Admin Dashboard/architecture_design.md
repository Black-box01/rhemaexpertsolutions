Monorepo-style Next.js App Router project rooted at `app/`. Public-facing routes live under `app/{coding-classes,competition,professional-trainings,verification-success}/page.tsx` plus the home page; shared chrome (`Header`, `HeroSlideshow`, `ImageGrid`, etc.) is in `components/`. Server-side data mutations are isolated in `app/actions/*.ts` (admin, auth, notes, registration, coding-classes) marked `'use server'` and invoked from client components via `revalidatePath`. Authentication is cookie-based: `app/actions/auth.ts` sets/deletes an `rhema_admin_auth` httpOnly cookie after validating against the `rhema_content` table's `admin.password` key (falling back to `ADMIN_PASSWORD` env). All Supabase access goes through `lib/supabase-admin.ts` (service-role client); `lib/supabase.ts` is kept separate for public reads. E-mail notifications are sent server-side through `lib/email.ts` (Nodemailer + Gmail SMTP), triggered on new competition/coding-class/professional-training registrations and on new staff e-notes. File attachments for e-notes use Supabase Storage bucket `staff-notes` via `uploadNoteFile`/`deleteNoteFile`; Cloudinary helpers in `lib/cloudinary.ts` and `scripts/upload-to-cloudinary.js` support legacy image uploads. The admin UI is a single large client component `app/admin/dashboard/page.tsx` that switches tabs over services/clients/team/competitions/newsletter/settings/registrations/coding-classes/staff-notes/professional-trainings, using local React state plus `useMemo` filtering and pagination driven by server actions. Database schema is versioned in `supabase_schema.sql` plus incremental migrations (`supabase_migration_*.sql`) and TypeScript types are generated into `types/supabase.ts`.
