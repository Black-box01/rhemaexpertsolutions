Built on Next.js App Router (`app/` directory) with a clear separation between public-facing pages (`app/page.tsx`, `app/coding-classes/page.tsx`, `app/professional-trainings/page.tsx`, `app/competition/page.tsx`) and an authenticated admin dashboard (`app/admin/dashboard/page.tsx`).

- Data layer: two Supabase clients — `lib/supabase.ts` (public anon client, read-only via RLS) and `lib/supabase-admin.ts` (service-role client bypassing RLS for admin writes). Types are centrally defined in `types/supabase.ts` and imported by both client and server code.
- Server actions: all mutations live in `app/actions/*.ts` (`admin.ts`, `auth.ts`, `notes.ts`, `registration.ts`, `coding-classes.ts`). Each action starts with `checkAuth()` from `app/actions/auth.ts` before touching the database, then calls `revalidatePath('/admin/dashboard')` after mutations.
- E-note system: `app/actions/notes.ts` provides paginated `fetchStaffNotes` (with `ilike` search over title/content plus status/category/priority filters), `saveStaffNote` (insert/update with email notification via `lib/email.ts`), `deleteStaffNote`, and file upload/delete against the Supabase Storage bucket `staff-notes` under `attachments/`. The admin dashboard owns the full UI state (pagination, search, drag-and-drop upload, confirmation dialogs).
- Shared UI: reusable React components in `components/` (`Header`, `HeroSlideshow`, `ImageWithSkeleton`, `AutoScrollGallery`, `NewsTicker`, etc.) consumed by pages; static assets live under `public/img/<category>/`.
- Database schema is versioned as SQL migrations at the repo root (`supabase_schema.sql`, `supabase_migration_*.sql`) and typed into `types/supabase.ts`.
